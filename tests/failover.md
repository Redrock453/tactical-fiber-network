# TFN Failover Testing — Сценарії обривів

## Огляд

Цей документ описує конкретні сценарії тестування живучості mesh-мережі TFN. Кожен сценарій має чіткі кроки та очікувані результати.

---

## Загальна вимога

> **Мережа витримує 1-2 обриви без втрати зв'язку між будь-якими вузлами.**

---

## Передумови

- 4+ вузлів у ring-топології
- BATMAN-adv налаштований на кожному вузлі
- Всі вузли пінгують один одного
- iperf3 встановлено на всіх вузлах

---

## Сценарій 1: Single Cable Cut (один обрив)

### Топологія

```
    A ═══════ B        A ═══════ B
    ║         ║   →    ║         ║  (обрив B→C)
    ║         ║        ║         ╳
    D ═══════ C        D ═══════ C
```

### Кроки

```bash
# 1. Записати початкові маршрути
batctl o > /tmp/routes_before.txt

# 2. Запустити безперервний ping
ping 192.168.10.3  # A → C

# 3. Розрізати кабель B→C (фізично)

# 4. Зафіксувати результати
```

### Метрики

| Метрика | Pass | Warning | Fail |
|---------|------|---------|------|
| Втрачені пакети | < 5 | 5-15 | > 15 |
| Час відновлення | < 5 сек | 5-10 сек | > 10 сек |
| Новий маршрут | A→D→C | A→D→C (повільно) | Зв'язок втрачено |
| Throughput після | > 80% від початкового | 50-80% | < 50% |

### Очікуваний результат

```
Before: A → B → C (2 hops, ~1ms)
After:  A → D → C (2 hops, ~1ms)
Lost packets: 3-5
Recovery time: 3-5 seconds
```

## Сценарій 2: Node Down (вузол offline)

### Топологія

```
    A ═══════ B        A ═══════ B
    ║         ║   →    ║         ║
    ║         ║        ║         ║
    D ═══════ C        D ═══════ C
                         ↑
                       ВИМКНЕНО
```

### Кроки

```bash
# 1. Запустити ping A→C та A→B

# 2. Вимкнути живлення вузла C

# 3. Зафіксувати:
#    - скільки часу до виявлення
#    - чи працює A→B→...→D
#    - batctl o (нові маршрути)

# 4. Вімкнути C назад

# 5. Зафіксувати час повернення
```

### Метрики

| Метрика | Pass | Warning | Fail |
|---------|------|---------|------|
| Виявлення C offline | < 5 сек | 5-10 сек | > 10 сек |
| Зв'язність A→B→D | Збережена | Частково | Втрачена |
| Повернення C | < 10 сек | 10-30 сек | > 30 сек |

## Сценарій 3: Double Cable Cut (два обриви)

### Топологія

```
    A ═══════ B        A ═══════ B
    ║         ║   →    ╳         ║
    ║         ║        ║         ╳
    D ═══════ C        D ═══════ C
```

### Кроки

```bash
# 1. Розрізати A→D
# 2. Розрізати B→C
# 3. Перевірити зв'язність:
#    - A→B (має працювати)
#    - C→D (має працювати)
#    - A→C (МАЄ ПРАЦЮВАТИ через A→B→...→C → НІ! Ring розділився!)
```

### Результат для ring

```
Ring з 4 вузлів + 2 обриви = МЕРЕЖА РОЗДІЛИЛАСЬ

A ←→ B ←→ (обрив) ←→ C ←→ D ←→ (обрив) ←→ A

Рішення: потрібен partial mesh (хоча б 1 діагональний лінк)
```

### Метрики

| Метрика | Ring | Partial Mesh |
|---------|------|-------------|
| 2 обриви витримує | НІ | **Так** |
| Потрібно додаткових лінків | 0 | 1-2 |
| Частка мережі, що працює | 50% | **100%** |

## Сценарій 4: Load + Cut (навантаження + обрив)

### Кроки

```bash
# 1. Запустити iperf3 A→C
iperf3 -c 192.168.10.3 -t 60 -b 100M

# 2. Під час передачі розрізати кабель

# 3. Зафіксувати:
#    - пропускна здатність до обриву
#    - пропускна здатність після відновлення
#    - скільки секунд простій
```

### Метрики

| Метрика | Pass | Warning | Fail |
|---------|------|---------|------|
| Throughput до обриву | > 900 Mbps | 500-900 | < 500 |
| Throughput після | > 700 Mbps | 300-700 | < 300 |
| Простой | < 5 сек | 5-10 сек | > 10 сек |

## Сценарій 5: Battery Depletion

### Кроки

```bash
# 1. Підключити вузол B через powerbank
# 2. Вимкнути зовнішнє живлення
# 3. Стежити за часом роботи
# 4. При ~10% — перевірити алерти
# 5. При 0% — перевірити failover
```

### Очікуваний результат

```
Powerbank 20Ah × 5V = 100Wh
Споживання вузла: ~5W
Час роботи: ~20 годин

При вимкненні B:
  - BATMAN перенаправляє трафік за 3-5 сек
  - A→D→C замість A→B→C
  - Алерт: "Node B offline (battery)"
```
