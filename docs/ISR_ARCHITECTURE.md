# TFN ISR System Architecture — SpiderLink

> "Лінія волокна = невидима стіна сенсорів + канал передачі + інструмент прийняття рішень."

---

## Мета системи

Побудувати **лінійну систему раннього виявлення** з безпечною передачею даних та підтримкою прийняття рішень оператором:

1. **Виявити** — сенсори фіксують рух, техніку, постріли вздовж лінії волокна
2. **Класифікувати** — edge-вузол визначає тип цілі та рівень загрози
3. **Передати** — дані йдуть через оптоволоконну mesh-мережу (0 RF-випромінювання)
4. **Візуалізувати** — C2 відображає події на карті в реальному часі
5. **Діяти** — оператор приймає рішення на основі підтверджених даних

**Ключове обмеження:** система повністю пасивна — жодного радіовипромінювання, жодного демаскуючого сигналу.

---

## Архітектурні рівні

Система складається з 4 рівнів, кожен з яких функціонує незалежно при відмові вищих рівнів.

```
┌─────────────────────────────────────────────────────────────────────┐
│                    LAYER 4: C2 CORE                                  │
│   FastAPI server + Operator UI + Event DB + Strike coordination     │
├─────────────────────────────────────────────────────────────────────┤
│                    LAYER 3: BACKBONE FIBER                           │
│   Ring + cross-links | BATMAN-adv | WireGuard | Redundant routes   │
├─────────────────────────────────────────────────────────────────────┤
│                    LAYER 2: EDGE NODES (10-20)                       │
│   Sensor node | Relay node | DAS node | Filter + Classify + Alert   │
├─────────────────────────────────────────────────────────────────────┤
│                    LAYER 1: SENSOR LAYER                             │
│   Piezo (MVP) → ADS1256 → ESP32 → UART/USB → Edge node             │
│   DAS (future) → φ-OTDR interrogation                               │
│   RF-Opto (future) → passive RF detection via fiber                 │
└─────────────────────────────────────────────────────────────────────┘
```

### Layer 1: Sensor Layer

Фізичні сенсори, що перетворюють вібрації/акустичні хвилі на цифровий сигнал.

#### Типи сенсорів

| Тип | Статус | Принцип | Чутливість | Вартість/канал |
|-----|--------|---------|-----------|----------------|
| **Piezo (MVP)** | Готовий до розгортання | П'єзокерамічний диск + LM358 + ADS1256 | 3-50м від кабелю | $1-5 |
| **Fiber DAS** | Потребує φ-OTDR | Rayleigh backscatter, фазочутливий аналіз | 1-80м від кабелю | $500-2000 (інтеррогатор) |
| **RF-Opto** | Дослідження | Ефект Керра + термооптичний ефект | 2-500м від RF-джерела | $0 (дод. обладнання не потрібне) |

#### Piezo sensor (MVP) — детально

```
[П'єзо 27mm] → [LM358 amp] → [ADS1256 24-bit ADC] → [ESP32-S3] → [UART/USB] → [Edge Node]
     ×4-8 шт       ×2 шт         30k SPS                  115200 baud        RPi/BPI-R3
```

- Кожен сегмент 50-100м: 1-2 п'єзо-сенсори
- Підключення через мультиплексор CD74HC4067 (до 16 каналів)
- Частотний діапазон: 0.1-500 Гц
- Частота дискретизації: 1000 SPS (можна до 10000)

**Живлення:** паразитне від edge-вузла (USB/DC), окреме живлення не потрібне.

#### Fiber DAS (майбутнє)

- Лазерний імпульс → волокно → backscatter → фотоприймач → ADC
- Просторова роздільна здатність: 1-5м
- Дальність: 10-40км по одному волокну
- Чутливість: 10⁻⁶ рад фазового зсуву
- Перевага: кабель сам є сенсором, ніяких додаткових приладів вздовж траси

#### Діаграма підключення (MVP)

```
                          Edge Node (RPi/BPI-R3)
                              │
                         USB/UART
                              │
                         [ESP32-S3]
                              │
                         SPI (ADS1256)
                              │
                    ┌─────────┼─────────┐
                    │         │         │
                [П'єзо CH0] [П'єзо CH1] [П'єзо CH2] [П'єзо CH3]
                 seg_01a     seg_01b     seg_02a      seg_02b
                    │         │         │             │
                 ───┴─────────┴─────────┴─────────────┴───
                          Волоконна траса (сегменти)
```

### Layer 2: Edge Nodes (10-20)

Обчислювальні вузли на передовій, що забезпечують класифікацію, маршрутизацію та живлення сенсорів.

#### Правила розміщення

**НЕ** рівномірно! Вузли розміщуються за тактичною доцільністю:

| Правило | Пояснення |
|---------|-----------|
| На поворотах траси | Точка максимального покриття (2 напрямки) |
| На перехрестях шляхів | Контроль кількох маршрутів одночасно |
| На критичних точках | Мости, вузькі місця, підходи до позицій |
| Інтервал 100-300м між вузлами | Залежить від рельєфу та загроз |
| Кожен вузол покриває 1-2 сегменти | Сегмент = 50-200м волокна |
| Вузол зміщений 2-5м від лінії кабелю | Anti-tracing (див. architecture/node_v2.md) |

#### Типи вузлів

| Тип | Функція | Компоненти | Вартість | Кількість у мережі |
|-----|---------|-----------|----------|-------------------|
| **Sensor node** | Сенсор + обчислення + fiber | BPI-R3 + SFP×2 + п'єзо + ADC + ESP32 + батарея | $210-283 | 8-12 |
| **Relay node** | Тільки маршрутизація fiber | BPI-R3 + SFP×2 + батарея | $169-243 | 3-5 |
| **DAS node** | Має φ-OTDR інтеррогатор | BPI-R3 + SFP×2 + φ-OTDR + батарея | $700-2200 | 1-2 |

#### Специфікація вузла (Sensor node, з node_v2.md)

```
Вузол v0.2 — "ROCK" (рознесена архітектура)

[Батарея LiFePO4 12V 20Ah]     ← закопана 10-20 см, 2-5 м від compute
        │
     3м DC кабель
        │
[Compute + SFP]                 ← замасковано, НЕ на лінії кабелю
   ┌──────────────┐
   │ Banana Pi    │             ← алюмінієва пластина знизу = радіатор
   │ BPI-R3       │
   │ 2× SFP       │
   │ WireGuard    │
   │ BATMAN-adv   │
   └──────────────┘
      │         │
   [Fiber IN] [Fiber OUT]      ← підходять З РІЗНИХ боків
```

#### Вартість edge-вузлів

| Компонент | Ціна |
|-----------|------|
| Banana Pi BPI-R3 | $80-120 |
| SFP модуль ×2 | $6-16 |
| Алюмінієва пластина 2мм | $3-5 |
| Термо-паста | $1 |
| Камуфляжна стрічка | $2 |
| MicroSD 32GB | $5-8 |
| LiFePO4 12V 20Ah | $60-80 |
| DC-DC 12V→5V USB-C | $5 |
| DC кабель 3м | $2 |
| Швидкі SC-коннектори ×2 | $4 |
| Гофрована трубка + epoxy | $2 |
| **РАЗОМ (sensor node)** | **$210-283** |
| **РАЗОМ (relay node, без сенсорів)** | **$169-243** |

#### BOM на 15 вузлів

| Категорія | Кількість | Вартість |
|-----------|-----------|----------|
| Sensor nodes | 10 | $2,100-2,830 |
| Relay nodes | 4 | $676-972 |
| DAS node (×1) | 1 | $700-2,200 |
| **Edge Nodes всього** | **15** | **$3,476-6,002** |

#### Функції edge-вузла

1. **Signal acquisition** — прийом даних від п'єзо-сенсорів (UART/USB)
2. **Filtering** — DC removal + band-pass (1-250 Гц) + Kalman
3. **FFT + Feature extraction** — RMS, peak frequency, spectral centroid, band energies
4. **Classification** — rule-based (MVP) → KNN/SVM (v0.2) → Conv1D (v0.3)
5. **Event generation** — JSON-подія з класифікацією, координатами, confidence
6. **Routing** — BATMAN-adv + WireGuard для передачі на C2
7. **Local autonomy** — при втраті зв'язку з C2: локальне журналювання + beacon сигнал

### Layer 3: Backbone Fiber

Оптоволоконна магістраль — єдиний канал передачі даних (RF заборонено).

#### Топологія

```
Ring + cross-links (мінімум для живучості):

     ●───●───●───●───●
     │               │
     ●   ╲     ╱    ●
     │      ╳       │
     ●   ╱     ╲    ●
     │               │
     ●───●───●───●───●
         │       │
         ●───●───●
              │
             C2

● = edge node    ─── = fiber link    ╳ = cross-link
```

#### Правила побудови магістралі

| Правило | Пояснення |
|---------|-----------|
| **Мінімум 2 маршрути до C2** | Жодна точка відмови не повинна ізолювати C2 |
| **Ring + 2 cross-links** | Витримує 2 одночасні обриви без втрати зв'язності |
| **"C2 ніколи не має єдиного кабелю"** | Обов'язкове дублювання підключення до C2 |
| **Хаотична прокладка** | Кабель НЕ по прямих лініях — anti-pattern-analysis |
| **Фальш-кабелі** | 1-2 пусті лінії на вузол для відволікання |
| **Глибина 10-20см** | Захист від виявлення, броньований на критичних ділянках |

#### Оцінка об'єму волокна

| Параметр | Значення |
|----------|----------|
| 15 вузлів, середня відстань 200м | ~3 км основного волокна |
| Cross-links (2 шт по ~400м) | ~0.8 км |
| Фальш-кабелі (15 × 15м) | ~0.2 км |
| З'єднувальні pigtail-и | ~0.2 км |
| Резерв (10%) | ~0.4 км |
| **РАЗОМ** | **~4.5-5 км** |

#### Протоколи

| Шар | Протокол | Функція |
|-----|----------|---------|
| L1 | SFP 1310nm 20km duplex LC | Фізичне з'єднання |
| L2 | BATMAN-adv | Mesh-маршрутизація, auto-failover |
| L3 | WireGuard (wg0) | Шифрування всього трафіку |
| L4 | JSON over UDP/TCP | Event повідомлення |
| L5 | WebSocket | Streaming до Operator UI |

#### Failover поведінка

```
Нормальний стан:
  Node_01 → Node_02 → ... → C2 (оптимальний маршрут)

Обрив Node_05 ↔ Node_06:
  BATMAN-adv виявляє: 3-5 секунд
  Новий маршрут: Node_04 → cross-link → Node_09 → Node_08 → ... → C2
  Мережа продовжує працювати

Другий обрив (Node_03 ↔ Node_04):
  BATMAN переобчислює: 5-10 секунд
  Другий cross-link забезпечує зв'язність
  Пропускна здатність падає, але система працює
```

### Layer 4: C2 Core

Командний пункт — центр управління, агрегації та візуалізації.

#### Апаратне забезпечення

| Компонент | Специфікація | Вартість |
|-----------|-------------|----------|
| Основний комп'ютер | Ноутбук або Mini-PC (Intel N100+) | $150-300 |
| Резервний комп'ютер | Той самий образ системи, ready to swap | $150-300 |
| Резервне живлення | UPS або LiFePO4 50Ah | $80-150 |
| Дисплей | Ноутбук-вбудований або зовнішній 15"+ | $0-100 |
| Зв'язок з мережею | 2× SFP підключення (дублювання) | $15-30 |

**Розташування:** 500м-2км за лінією сенсорів, у захищеному бліндажі/укритті.

#### Програмне забезпечення

```
C2 Server (FastAPI)
├── /events          — POST: прийом ISR-подій від edge-вузлів
├── /events          — GET:  історія подій (фільтрація, пагінація)
├── /events/stream   — WebSocket: реальний час для Operator UI
├── /status/nodes    — статус всіх edge-вузлів (health, battery)
├── /status/links    — статус fiber-лінків (BATMAN-adv)
├── /action/strike   — POST: запит на ураження цілі
├── /action/ack      — POST: підтвердження дії оператором
└── /config          — оновлення конфігурації (поріг, правила)

Operator UI (HTML + JS, одинарний файл)
├── Тактична карта (Leaflet.js)
│   ├── Позиції вузлів
│   ├── Fiber траса
│   ├── Сенсорні сегменти
│   └── Активні події (кольорове кодування)
├── Панель подій
│   ├── Реальний час (WebSocket)
│   ├── Фільтрація: тип, загроза, confidence
│   └── Історія з пошуком
├── Панель дій
│   ├── "Запит FPV" → вибір цілі → оператор підтверджує
│   ├── "Викликати артилерію" → координати автоматично
│   └── "Позначити хибне спрацьовування" → ML-навчання
└── Моніторинг системи
    ├── Вузли: online/offline, battery, uptime
    ├── Fiber: SNR, attenuation per link
    └── Статистика: подій/год, detection rate, false alarm rate
```

#### Функції C2

| # | Функція | Опис |
|---|---------|------|
| 1 | **Event aggregation** | Прийом ISR-подій від всіх edge-вузлів, дедуплікація |
| 2 | **Filtering** | Видалення шуму, дублікатів, хибних спрацьовувань |
| 3 | **Visualization** | Карта + список подій + статус мережі |
| 4 | **Prioritization** | Сортування за рівнем загрози (critical → high → medium → low) |
| 5 | **Action interface** | Кнопка "Ураження" → оператор підтверджує → координати цілі |
| 6 | **Track correlation** | Об'єднання подій з різних сегментів у трек цілі |
| 7 | **BDA (Battle Damage Assessment)** | Після ураження сенсор підтверджує результат |

## Data Flow Diagrams

### Нормальна робота

```
Sensor → [Edge Node] → [Fiber Mesh] → [C2 Server] → [Operator UI]
  piezo    filter/classify    BATMAN-adv    FastAPI       Browser
  ADC      FFT+features       WireGuard     WebSocket     Alerts
  1000SPS  event JSON         ring route    events DB     Strike btn
  <10ms    <100ms             <1ms/km       <50ms         <100ms
```

### C2 disconnected (edge autonomy)

```
Sensor → [Edge Node] → [Local Alert] → [Beacon]
  piezo    filter/classify    log event       flash/LED
  ADC      confidence > 0.8   queue for C2    сигнал nearby units
```

Edge-вузли продовжують класифікувати події навіть при втраті зв'язку з C2. Події буферизуються і передаються при відновленні зв'язку.

## Message Format (ISR Event)

Стандартизований JSON-формат події, що використовується на всіх рівнях системи.

### Схема події

```json
{
  "event_id": "evt_20260419_034512_001",
  "timestamp": "2026-04-19T03:45:12.345Z",
  "source": {
    "node_id": "node_03",
    "sensor_type": "piezo",
    "sensor_channel": 0,
    "segment_id": "seg_03",
    "position": {
      "fiber_m": 650,
      "lat": 48.5123,
      "lon": 36.3456
    }
  },
  "classification": {
    "target": "tracked_vehicle",
    "confidence": 0.87,
    "threat_level": "high",
```

## Integration Points

### Level 1: Manual (MVP)

Повністю ручний цикл, система — помічник оператора.

```
Система надає:       Оператор робить:
  сектор (seg_03)    → оцінює ситуацію
  час (14:23:15)     → приймає рішення
  тип цілі (танк)    → обирає зброю
  confidence (87%)   → запускає FPV / викликає артилерію
```

**Переваги:** Мінімум автоматизації, оператор завжди контролює.
**Недоліки:** Повільний цикл (30-120 сек від виявлення до дії).

### Level 2: Semi-automatic

Система рекомендацій — оператор підтверджує.

```
Система рекомендує:
  "Гусенична техніка в seg_03, рухається на південний захід, швидкість 15 км/год"
  "Рекомендоване ураження: FPV, координати [48.5118, 36.3461]"
  "Час до втрати цілі з сектору: ~90 секунд"

Оператор:
  Натискає "Підтвердити" → FPV автоматично направляється в сектор
  Або "Відхилити" → подія логується як відхилена
```

**Переваги:** Швидший цикл (10-30 сек), зменшення навантаження на оператора.
**Недоліки:** Потрібна stable detection + reliable comm.

### Level 3: Full integration (майбутнє)

Автоматичне цілевказування на основі трекового прогнозування.

```
Система автоматично:
  1. Виявляє ціль у seg_02
  2. Трекує рух через seg_03 → seg_04
  3. Прогнозує позицію через 30 сек
  4. Призначає FPV з найближчої стартової позиції
  5. Передає координати автоматично

Оператор:
  Тільки контролює ROE (Rules of Engagement)
  Підтверджує або скасовує автоматичні призначення
```

**Вимоги:**
- Stable detection rate > 80%
- Надійний зв'язок (fiber mesh без деградації)
- Налаштовані ROE (зони заборони вогню, цивільні об'єкти)
- Backup: система завжди питає підтвердження при uncertainty > 0.3

## Latency Budget

| Етап | Latency | Примітка |
|------|---------|----------|
| Sensor → Edge Node (UART/USB) | < 10 ms | ESP32 → RPi, 115200 baud |
| ADC sampling + buffering | 10-50 ms | 1024 samples @ 1000 SPS |
| Edge classification (rule-based) | < 50 ms | FFT + features + rules |
| Edge classification (ML) | < 100 ms | SVM/KNN на BPI-R3 |
| Edge → C2 (fiber, per km) | < 1 ms | BATMAN-adv + WireGuard overhead |
| C2 processing + DB write | < 50 ms | FastAPI + SQLite/JSON |
| C2 → Operator (WebSocket) | < 100 ms | Browser rendering |
| **TOTAL: sensor → operator** | **< 500 ms** | **Оптимальний маршрут** |
| **TOTAL: 15-hop маршрут** | **< 800 ms** | **Найгірший випадок** |

|---------|----------------------------|----------|
| **TFN SpiderLink** | **< 500 ms** | Fiber, passive |

| Камери + ML | 1-5 сек | Video frame processing |
| Патруль / спостерігач | 5-30 хв | Людський фактор |

```
[ ] Повторити для вузлів 02-15

[ ] Запустити FastAPI server
[ ] Перевірити: отримання events від всіх вузлів
[ ] Відкрити Operator UI в браузері
[ ] Тестовий alert від кожного вузла

ЕТАП 5: Калібрування [1-2 години]
```

```
[ ] Всі 15 вузлів online в batctl n

[ ] Operator UI працює без затримок

```
