# TFN Edge Node v0.2 — Специфікація фізичної живучості

> "Найкращий вузол — той, якого не знайшли."  
> "Другий найкращий — той, який замінили за 2 хвилини."

---

## 1. Філософія живучості

### Вузли — витратний матеріал (consumables)

Вузли **БУДУТЬ** знищуватись. Це не питання "якщо", а питання "коли". Архітектура будується на цьому допущенні.

**Мета:** мінімізація шансу виявлення (detection probability), а не повна невидимість. Повна невидимість — міф. Реальна мета — бути складнішим для виявлення, ніж вартість зусиль по пошуку.

### Моделір загроз

| Загроза | Рівень | Контрзахід |
|---------|--------|------------|
| RF emission detection (РЕР) | НИЗЬКИЙ | 0 RF emission — passive optical |
| Візуальне виявлення | ВИСОКИЙ | Камуфляж, неконтролерована форма |
| Теплова сигнатура (thermal) | СЕРЕДНІЙ | Рознесення, контакт з ґрунтом |
| Патерн-аналіз прокладки | ВИСОКИЙ | Хаотична прокладка, зміщення вузлів |
| Фізичне трасування кабелю | ВИСОКИЙ | Зміщення вузла, фальш-лінії |
| Захоплення вузла (capture) | КРИТИЧНИЙ | WireGuard, ротація ключів |

### Сильні сторони

- **0 RF emission** → не бачить радіоелектронна розвідка (РЕР)
- Passive optical мережа → немає радіовипромінювання взагалі
- Низьке енергоспоживання → мала теплова сигнатура
- Дешевий вузол (<$200) → швидка заміна економічно доцільна

### Слабкі сторони

- **Фізична видимість** — об'єкт існує в матеріальному світі
- **Теплова сигнатура** — навіть 3W дають ΔT над навколишнім середовищем
- **Патерни прокладки** — прямі лінії кабелю читабельні з повітря
- **Трасування кабелю** — знайшов кабель → йди по ньому → знайди вузол

## 2. Фізична схема вузла v0.2

### Вузол v0.2 — "ROCK" (рознесена архітектура)

```
Вузол v0.2 — "ROCK" (рознесена архітектура)

[Батарея LiFePO4]     ← закопана 10-20 см, 2-5 м від обчислювача
       │
    3м кабель (DC)
       │
[Compute + SFP]        ← замасковано, НЕ на лінії кабелю
  ┌─────────────┐      ← зсунутий вбік 2-5 м від fiber-лінії
  │ Banana Pi   │
  │ BPI-R3      │      ← алюмінієва пластина знизу = радіатор + маскування
  │ 2× SFP      │
  │ WireGuard   │
  └─────────────┘
     │         │
  [Fiber IN] [Fiber OUT]  ← підходять З РІЗНИХ боків, не пряма лінія
```

### Принцип рознесення (separation principle)

```
ВИГЛЯД ЗВЕРХУ (не в масштабі):

    Fiber A                        Fiber B
       ~~~~                           ~~~~
           ~~~~        ┌──────┐   ~~~~
                ~~~~~~~│ ВУЗОЛ│~~~
                       └──────┘
                           │
                        DC кабель
                           │
                    ┌──────────────┐
                    │  БАТАРЕЯ     │
                    │  (закопана)  │
                    └──────────────┘

    ← 2-5м →  ← 2-5м →  ← 2-5м →
    
    Fiber A ←~~~~~~~→ Вузол ←~~~~~~~→ Fiber B
                 зміщення    зміщення
                 2-5м вбік   2-5м вбік
```

### Розміри компонентів

```
Compute block:    ~12×8×3 см (без корпусу)
З алюмінієвою пластиною: ~14×10×3.5 см
Вага compute:     ~0.3 кг
Батарея LiFePO4 20Ah: ~18×8×7 см
Вага батареї:     ~1.5 кг
DC кабель:        3м, 2×0.75мм²
Загальна вага:    ~2 кг
```

## 3. Маскування (Concealment)

### 3.1 Візуальне маскування (Visual Concealment)

**Принцип:** "Не виглядати як техніка." Людський мозок шукає знайомі патерни: прямокутники, LED, антенки, кабелі.

| Правило | Опис | Чому |
|---------|------|------|
| **ЖОДНИХ LED** | Фізично видалити або заклеїти чорною стрічкою + epoxy | LED видно вночі за 100+м, особливо з ПНВ |
| **Корпус: чорний / хакі / "камінь"** | Камуфляжна стрічка, матова фарба, texture spray | Блиск = помітно. Матове = природно |
| **Форма: НЕ прямокутна** | Заокруглені краї, asymmetric shape, "natural" outline | Прямокутник = "техніка" в мозку спостерігача |
| **Матеріал: алюміній з матовим покриттям** | Алюмінієва пластина 2мм + матова фарба / camo tape | Блискучий метал = демаскуючий ознака |
| **Вигляд = елемент місцевості** | Камінь, гілка, корч, сміття, будівельне сміття | Злиття з фоном (background matching) |

### 3.2 Теплове маскування (Thermal Concealment)

**Принцип:** "Розмазати тепло, а не сховати." Неможливо повністю прибрати теплову сигнатуру. Мета — зменшити ΔT до рівня, коли тепловізор не відрізнить вузол від навколишнього середовища.

**Тепловий бюджет Banana Pi BPI-R3:**

| Режим | Споживання | ΔT над навколишнім | Помітність |
|-------|-----------|---------------------|------------|
| Idle (базова маршрутизація) | ~3W | ~5-8°C | НИЗЬКА з 10+м |
| Load (DAS processing) | ~5-7W | ~10-15°C | СЕРЕДНЯ з 20+м |
| Peak (boot, update) | ~10W | ~15-20°C | ВИСОКА з 50+м |

**Контрзаходи:**

1. **НЕ герметичний корпус** — теплорозсіювання через контакт з землею
   - Герметичний = heat trap = яскрава точка на thermal imager
   - Відкритий + contact з ґрунтом = розподіл тепла на велику площу
   
2. **Алюмінієва пластина знизу як радіатор**
   - Контакт з ґрунтом через thermal paste
   - Площа розсіювання >> площа вузла
   - Ґрунт = величезний heat sink (~10°C/м стабільної температури)
   
3. **Рознесення компонентів**
   - Батарея окремо → тепловий signature розділений на 2 менші точки
   - Замість 1 точки ~5W → 2 точки ~3W + ~2W
   - 2 маленькі точки менш помітні ніж 1 велика
   
4. **Безпечна відстань: 2-5м між компонентами**
   - Достатньо для розділення на thermal imager
   - Практично для польового розгортання

```
Тепловий профіль (вид зверху, thermal imager):

   БЕЗ маскування:          З маскуванням:
   
   ■■■■■■■■                 ·  ·
   ■■■■■■■■               (ледь помітні
   ■■■■■■■■                точки, < 2°C
   (яскравий об'єкт,        над фоном)
    ΔT = 15°C)
```

### 3.3 Патерн-аналіз (Pattern Analysis Countermeasures)

**Загроза:** Аналіз з повітря (drone / satellite) виявляє патерни прокладки кабелю → прямі лінії = технічна інфраструктура.

| Загроза | Контрзахід |
|---------|------------|
| Прямі лінії кабелю | Кабель НЕ по прямих лініях: A ~~~ B (хаотична прокладка, "змійка") |
| Вузол на лінії кабелю | Вузол НЕ на лінії кабелю: зміщений вбік 2-5м |
| Кабелі підходять з одного напрямку | Fiber підходить з різних напрямків (кут ≥90°) |
| Регулярні відстані | Варіювати відстані між вузлами (200-800м) |
| Одиночний кабель | Фальш-кабелі: пусті лінії для відволікання |

## 4. Anti-Tracing (Захист від трасування)

### Принцип "розриву логіки"

Обробник, який знайшов кабель, намагається йти по ньому до вузла. Мета — зробити цей процес максимально складним.

```
АНТИ-ТРАСУВАННЯ:

    Кабель знайдено ──→ йдемо по кабелю ──→ ──→
                                              │
                                         точка розриву
                                         (кабель змінює напрямок
                                          або зникає під землею)
                                              │
                                    ┌─────────┴────────┐
                                    │                   │
                               фальш-лінія ←── ──→ ВУЗОЛ (2-5м вбік)
                               (порожній кабель)     замасковано
```

### Правила anti-tracing

1. **"Розрив логіки":** кабель → (2-5 м) → вузол збоку
   - По кабелю НЕ можна вийти прямо на вузол
   - Вузол зсунутий відносно осі кабелю
   
2. **Зміщення вузла:**
   - Вузол НЕ на лінії кабелю
   - Offset: 2-5м перпендикулярно до осі кабелю
   - Підключення через короткий pigtail що йде вбік
   
3. **Фальш-кабелі (decoy lines):**
   - Порожні відрізки fiber-кабелю (5-20м)
   - Прокладені в іншому напрямку від точки підключення
   - Збільшує час пошуку: знайшов кабель → йде по фальш-лінії → тупік
   - Мінімум 1 фальш-кабель на кожен вузол
   
4. **Глибина закладання:**
   - Вузол: поверхня або 2-3 см (під маскуванням)
   - Кабель: 5-10 см мінімум, 10-20 см краще
   - При підході до вузла кабель заглиблюється → важче знайти точку виходу

```
ПЕРЕРІЗ (бічний вид):

  Поверхня ──────────────────────────────────
       ↓ маскування (листя, гілки, каміння)
       ↓ ┌──────────┐
       ↓ │  ВУЗОЛ   │ ← на рівні поверхні або трохи нижче
       ↓ └────┬─────┘
       ↓      │ pigtail 2-5м вбік
       ↓      │
  -5см ↓ ─────┼──────────────────────────────
       ↓      │
  -10см↓      ╰──── кабель ──────────────────
       ↓
  -20см↓ ════════════════════════════════════  (глибше = краще)
```

## 5. Захист від захоплення (Capture Protection)

### Принцип: компрометація одного вузла ≠ компрометація мережі

### Криптографічний захист

| Захід | Реалізація |
|-------|------------|
| **WireGuard обов'язковий** | wg0 на кожному інтерфейсі, весь трафік зашифрований |
| **ЖОДНОГО відкритого LAN** | Всі порти = WireGuard tunnels або disabled |
| **Pre-shared keys (PSK)** | Ключі попередньо встановлені, додатковий рівень Authentication |
| **Ротація ключів** | При втраті вузла — ротація через наступне з'єднання |
| **Немає remote console** | Консольний доступ = лише фізичне підключення (serial / HDMI) |
| **No persistent logs** | Логи тільки в RAM, стираються при вимкненні |

### Архітектура WireGuard

```
[Fiber IN] ──→ SFP#1 ──→ wg0-in (encrypted tunnel)
                              │
                         [Routing Engine]
                         (BATMAN-adv)
                              │
[Fiber OUT] ←── SFP#2 ←── wg0-out (encrypted tunnel)
```

### Процедура при втраті вузла

1. Виявити втрату (сусіди не відповідають через batctl ping)
2. Відключити compromised links на сусідніх вузлах
3. Згенерувати нові ключі: `wg genpsk > /etc/wireguard/psk_peerX`
4. Розгорнути нові ключі на сусідніх вузлах через physical delivery
5. Встановити замінний вузол з новими ключами
6. Загальний час: < 30 хвилин при наявності запасного вузла

### Чого НЕ робити

- НЕ використовувати SSH з паролями
- НЕ залишати debug ports відкритими
- НЕ зберігати ключі інших вузлів на диску (тільки PSK для direct peers)
- НЕ мати auto-update (може бути вектором атаки)
- НЕ підключати до Internet без VPN

## 6. Заміна вузла за 2 хвилини (Hot-Swap Procedure)

### Принцип: plug-and-play в польових умовах

Підключив 2 коннектори + живлення = працює. **Нуль налаштувань в полі.** Все попередньо сконфігуровано на MicroSD/eMMC.

### Передумови

- Запасні вузли попередньо налаштовані (pre-configured image на MicroSD)
- WireGuard ключі попередньо встановлені для всіх expected peers
- BATMAN-adv сконфігурований на автозапуск
- Мінімум **2 запасні вузли** в готовності на кожні 10 активних

### Процедура заміни

```
ЗАМІНА ВУЗЛА — < 2 ХВИЛИНИ

  ┌─────────────────────────────────────────────────┐
  │ Крок 1: ВІДКЛЮЧИТИ ЖИВЛЕННЯ      [0:00-0:10]  │
  │   → Від'єднати DC коннектор від старого вузла   │
  │                                                  │
  │ Крок 2: ВІДКЛЮЧИТИ FIBER          [0:10-0:20]  │
  │   → Від'єднати 2 SC-коннектори                  │
  │   → (fiber кабелі залишаються в землі)          │
  │                                                  │
  │ Крок 3: ВИТЯГТИ СТАРИЙ ВУЗОЛ      [0:20-0:40]  │
  │   → Витягти з маскування                        │
  │   → Зберегти якщо можливо (repair)              │
  │                                                  │
  │ Крок 4: ВСТАВИТИ НОВИЙ ВУЗОЛ     [0:40-1:00]   │
  │   → Встановити на місце старого                 │
  │   → Замаскувати (parcially, швидко)             │
  │                                                  │
  │ Крок 5: ПІДКЛЮЧИТИ               [1:00-1:30]   │
  │   → 2 SC-коннектори (Fiber IN, Fiber OUT)       │
  │   → DC коннектор (живлення)                     │
  │                                                  │
  │ Крок 6: ПЕРЕВІРИТИ                [1:30-2:00]   │
  │   → batctl n (бачить сусідів)                   │
  │   → ping через WireGuard tunnel                 │
  │   → Якщо OK → фінальне маскування              │
  └─────────────────────────────────────────────────┘
```

### Checklist для запасного вузла

```bash
# Перевірити перед виходом в поле:
[ ] MicroSD з pre-configured image вставлена
[ ] WireGuard ключі актуальні
[ ] BATMAN-adv налаштований (batctl if addmesh bat0)
[ ] SFP модулі вставлені
[ ] SC-коннектори в комплекті (2 шт запасних)
[ ] Камуфляжна стрічка в комплекті
[ ] Батарея заряджена (якщо заміна батареї теж)
[ ] Алюмінієва пластина + thermal paste
[ ] Тестовий запуск: batctl n, wg show
```

## 7. Кабельна інфраструктура (Cable Infrastructure)

### Правила прокладки

| Параметр | Значення | Обґрунтування |
|----------|----------|---------------|
| **Прокладка** | НЕ пряма (хаотична, "змійка") | Анти-патерн-аналіз |
| **Глибина мінімум** | 5-10 см | Захист від випадкового виявлення |
| **Глибина рекомендована** | 10-20 см | Захист від intentional search |
| **Критичні ділянки** | Броньований кабель або подвійний | Перетини доріг, водних перепон |
| **Фальш-лінії** | 1-2 на вузол, пусті кабелі | Відволікання при трасуванні |
| **Мінімальний радіус вигину** | 30 мм (G.657.A2) | Запобігання макро-зігинання втрат |
| **Максимальна довжина сегмента** | 2000м (без сплайсів) | Обмеження SFP budget |

### Схема прокладки

```
ПРАВИЛЬНА прокладка (вид зверху):

    Вузол A ~~~ ~~~ ~~~ ~~~ ТУТ ~~~ ~~~ ~~~ Вузол B
                               │
                           фальш-кабель
                           (порожній, 10м)
                               │
                           тупік

    "~~~" = хаотична змійка (не пряма лінія)
    "ТУТ" = точка розриву логіки (кабель змінює напрям)


НЕПРАВИЛЬНА прокладка:

    Вузол A ──────────────────────────── Вузол B
    
    (пряма лінія = легко виявити з повітря і трасувати)
```

### Типи кабелю

| Тип | Застосування | Примітка |
|-----|-------------|----------|
| G.657.A2 simplex | Стандартні сегменти | Гнучкий, малий радіус вигину |
| Броньований simplex | Критичні ділянки | Захист від механічних пошкоджень |
| Фальш-кабель (будь-який) | Декой | Порожній, для відволікання |

### З'єднання (splicing)

- Швидкі SC-коннектори (field-installable) на кожному кінці
- Механічні сплайси для ремонту в полі
- Гофрована трубка для захисту сплайсів
- Ніякого fusion splicing в полі (занадто складно і довго)

## 8. Живучість мережі (Network Survivability)

### Принцип: graceful degradation

Мережа не падає повністю при втраті вузлів. Замість цього:

- **Падає bandwidth**, але працює
- **Падає кількість маршрутів**, але залишається зв'язність
- **DAS продовжує працювати** на збережених сегментах
- **Часткове покриття** краще ніж повна відсутність

### Топологічна надлишковість

```
RING TOPOLOGY (мінімум для живучості):

    A ──── B
    │      │
    │      │
    D ──── C

    Втрата 1 вузла: A─B─C─D (ланцюг, працює)
    Втрата 2 вузлів: A─B або C─D (сегменти, працюють окремо)


PARTIAL MESH (краща живучість):

    A ──── B
    │╲    ╱│
    │ ╲  ╱ │
    │  ╲╱  │
    D ──── C

    Втрата 1 вузла: 3 маршрути між іншими
    Втрата 2 вузлів: direct link між залишившимися
```

### Метрики живучості

| Метрика | Ціль | Метод |
|---------|------|-------|
| Швидкість заміни вузла | < 2 хв | Hot-swap procedure |
| Вартість замінного вузла | <$200 | BOM оптимізація |
| Запасні вузли в готовності | ≥2 на 10 активних | Логістика |
| Час відновлення зв'язності | < 5 хв | BATMAN-adv auto-reconverge |
| Час повного відновлення мережі | < 30 хв | Physical deployment |

### BATMAN-adv behavior при втраті вузла

```bash
# BATMAN-adv автоматично reconverges
# Таймаут за замовчуванням: 200 секунд

# Прискорити convergence:
```

## 10. Тест "5 хвилин" (Five-Minute Concealment Test)

### Призначення

Стандартизована процедура перевірки якості маскування вузла. Мінімальний acceptance test перед розгортанням.

### Процедура

```
ТЕСТ "5 ХВИЛИН"

Етап 1: Встановлення [10 хв]
  ┌─────────────────────────────────────────────────┐
  │ 1. Встановити вузол в реальному середовищі      │
  │ 2. Замаскувати згідно з процедурою              │
  │ 3. Повністю завершити маскування                │
  │ 4. Відійти на 50м від точки установки           │
  └─────────────────────────────────────────────────┘

Етап 2: Візуальний пошук [5-15 хв]
  ┌─────────────────────────────────────────────────┐
  │ 1. Попросити людину (не залучену до установки)  │
  │    знайти вузол в радіусі 20м від known area    │
  │ 2. Засікти час                                 │
  │ 3. Фіксувати результат:                        │
  │    - < 5 хв  → ПРОВАЛ (критичне маскування)     │
  │    - 5-15 хв → ПОТРІБНЕ ПОКРАЩЕННЯ              │
  │    - > 15 хв → УСПІХ                           │
  └─────────────────────────────────────────────────┘

Етап 3: Тепловізорна перевірка [5 хв]
  ┌─────────────────────────────────────────────────┐
  │ 1. Тепловізор (FLIR або аналогічний)           │
  │ 2. Порівняти область вузла з навколишнім фоном │
  │ 3. Критерій: ΔT < 3°C на відстані 10м          │
  │    - ΔT > 5°C  → ПРОВАЛ                        │
  │    - ΔT 3-5°C → ПОТРІБНЕ ПОКРАЩЕННЯ             │
  │    - ΔT < 3°C → УСПІХ                          │
  └─────────────────────────────────────────────────┘

Етап 4: Drone пошук (опціонально) [10 хв]
  ┌─────────────────────────────────────────────────┐
  │ 1. Drone з камерою на висоті 30-50м            │
  │ 2. Пошук візуально + тепловізорно               │
  │ 3. Критерій: не виявлено за 10 хв → УСПІХ      │
  └─────────────────────────────────────────────────┘
```

### Результати тесту (log template)

```
Дата:          _______________
Місцевість:    _______________ (ліс / поле / місто / змішана)
Погодні умови: _______________ (температура, вітер, хмарність)
Встановник:    _______________
Шукач:         _______________ (не бачив установку)

Візуальний пошук:
  Час знаходження: _____ хв
  Результат: ПРОВАЛ / ПОКРАЩЕННЯ / УСПІХ

Тепловізорна перевірка:
  ΔT виміряний: _____ °C
  Відстань виявлення: _____ м
  Результат: ПРОВАЛ / ПОКРАЩЕННЯ / УСПІХ

Drone пошук (якщо проводився):
  Виявлено: ТАК / НІ
  Час: _____ хв

Примітки: _______________________________________________
```

### Частота тестування

- Кожен новий тип маскування — обов'язковий тест
- При зміні середовища (ліс → місто) — повторний тест
- Мінімум 1 раз на місяць для активних вузлів
- Після будь-якого виявлення вузла противником — аналіз + повторний тест
